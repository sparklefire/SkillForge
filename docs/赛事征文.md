# SkillForge（匠传）：让Agent把专家经验锻造成可验证的培训包

> SkillForge把专家操作视频、设备手册和口述经验转化为带证据的SOP、检查清单、测验、海报和培训视频。它的重点不是“多生成一份内容”，而是让Agent完成“发现问题 → 展示证据 → 局部修订 → 再次验证”的闭环。

## 项目背景

在仓库、门店和小型制造现场，新员工学习设备操作时，知识往往散落在三种材料里：一段老师傅录下的视频、几十页设备手册，以及只有熟练操作员才知道的口述经验。视频容易看漏动作，手册不一定覆盖现场习惯，口述又很难追溯。如果只让大模型直接写教程，它可能漏掉关键步骤、打乱先后顺序，或者补出看似合理却没有来源的工具和参数。

SkillForge（中文名“匠传”）希望解决的不是文案效率问题，而是培训知识的可信度问题。我们要求每个关键步骤都能回到PDF页码、视频时间点或录音时间点；模型推断不能冒充设备事实，操作者经验也不能被混写成厂商规定。只有在证据边界清楚的前提下，Agent才能真正参与工业培训内容创作。

## SkillForge如何工作

系统先把视频、PDF和录音解析成统一的Evidence Catalog，再由Perception、SOP、Creator、Verifier和Revision五类Agent协作。当前公开追踪记录了五类Agent、13种工具和14次工具调用，并保留Agent交接、输入输出成果哈希与Evidence绑定。完整追踪可以在[Agent与工具调用报告](https://github.com/sparklefire/SkillForge/blob/main/cases/n31/evaluations/agent_tool_trace_v1.json)中复核。

Perception负责提取候选页、时间点和结构；SOP Agent规划步骤和依赖；Creator生成不同培训成果；Verifier检查缺步骤、顺序、工具、参数和来源；Revision只修改受影响单元。系统不会因为一句话读起来顺畅就接受它，而是要求必要工具、参数、警告和完成标准都落在当前步骤的Evidence边界内。

## 真实案例

主案例是“汉印N31电子面单打印机更换标签纸、介质学习与试印验收”。输入来自真实设备、自摄并经过本地隐私处理的安全视频、两份设备手册和实际操作者口述。最终形成13步Gold SOP，其中10步必需、3步按条件执行。Gold由实际操作者审核，不是模型自动给自己打分。步骤、证据和状态可在[Gold SOP](https://github.com/sparklefire/SkillForge/blob/main/cases/n31/gold/gold_sop.json)中查看。

两份手册共58页，形成607个页码绑定结构块；9页在本地完成中文OCR，待处理页为0。视频、手册和口述先分别产生18条、7条和8条候选，再经过粗粒度拆分、细粒度合并、同义项归并和依赖图检查，得到13个有序步骤。12步至少有两类来源，10步覆盖全部三类来源，详细结果见[PDF结构报告](https://github.com/sparklefire/SkillForge/blob/main/cases/n31/evaluations/pdf_structure_v1.json)和[来源候选合成报告](https://github.com/sparklefire/SkillForge/blob/main/cases/n31/evaluations/source_candidate_synthesis_v1.json)。

## Agent质检与局部修订

为了让演示可复现，我们没有在真实设备上故意制造危险操作，而是在操作者审核Gold的副本中注入受控错误：缺少介质学习、前置依赖断裂、步骤顺序错误、无依据参数和无依据工具。Verifier引用当前步骤证据后，检测到严重问题从5项降至0项的修订需求；Revision执行4项局部修订，包括插入缺失步骤、删除无依据参数、删除无依据工具和恢复正确顺序。必要步骤覆盖和证据覆盖都从90%提升到100%，结果保存在[闭环评测摘要](https://github.com/sparklefire/SkillForge/blob/main/cases/n31/demo_bundle/summary.json)和[修订审计](https://github.com/sparklefire/SkillForge/blob/main/cases/n31/demo_bundle/revision_audit.json)中。

“局部”不是宣传用语。选择性重建分析显示，这组冲突只影响7个步骤、1道题和7个视频镜头；无关步骤、题目和镜头保持不变。对应边界由[选择性重建报告](https://github.com/sparklefire/SkillForge/blob/main/cases/n31/evaluations/selective_rebuild_v1.json)逐项记录。这样既避免整包重写带来的新错误，也让审核者能明确看到修改前、问题、证据、修改后和复检结果。

## DGX Spark与技术实现

P0主链采用原生Python与FFmpeg，不依赖Docker。DGX Spark上的NVIDIA GB10使用CUDA 13原生内核处理6段安全派生视频、420帧、50个候选时间点，计算范围明确限定为场景候选筛选，不把亮度、边缘或变化分数写成操作事实。GPU设备、编译架构、来源策略和逐段统计都记录在[DGX视觉计算报告](https://github.com/sparklefire/SkillForge/blob/main/cases/n31/evaluations/dgx_visual_compute_v1.json)中。

DGX还承载完整离线Gold闭环和只监听回环地址的Web演示服务，并通过SSH隧道提供现场访问。Step 3.7语义规划和视觉复核使用Step Plan API；原始视频、手册页面、完整录音、本地路径和凭证不发送给外部模型。我们明确区分本地计算与外部模型推断，不把外部API包装成DGX本地推理。

## 可复现结果

当前真实Gold结果包括：严重问题5→0、4项局部修订、必要步骤与证据覆盖90%→100%。多源消融显示，手册单源覆盖80%，专家口述单源覆盖90%，至少两类来源联合覆盖100%；这说明真实培训知识并不适合只依赖单一材料。消融详情见[多源对比报告](https://github.com/sparklefire/SkillForge/blob/main/cases/n31/evaluations/multisource_comparison_v1.json)。

在DGX Python 3.12环境中，结构化Gold Web现场重算中位数为44.8毫秒，测量包含2次预热和20次正式运行。这个数值不包含原始视频、PDF或录音的预处理，也不代表整机GPU推理延迟；完整范围和样本见[DGX运行时基准](https://github.com/sparklefire/SkillForge/blob/main/output/evaluation/runtime_benchmark_dgx.json)。项目提供现场重算、预处理结果和无素材离线包三种演示模式，并为每种模式固定入口、数据边界和期望结果。

## 交付成果

同一份证据约束SOP继续生成13项手机检查清单、5题培训测验、一页A4海报和80秒培训视频。视频包含15个镜头覆盖13/13个Gold步骤，绑定25条去重Evidence；自动QA已经通过，但最终提交前仍保留参赛者完整观看和隐私复核。视频的时长、编码、覆盖率、来源策略和Evidence包哈希记录在[培训视频清单](https://github.com/sparklefire/SkillForge/blob/main/output/video/n31_training_video_manifest_v1.json)中。

除此之外，仓库还提供8页路演PPT、三种SOP视图、连续动作候选窗口、PDF结构报告、语义复核、无来源内容门禁、可恢复工作流、确定性技术交付包和原生Python Web Demo。公开仓库只保存经过审核的结构化结果与安全派生成果，原始行业素材、个人信息和提交表单私有状态不进入Git。

## 边界与下一步

SkillForge当前仍然需要人工审核，也不是全自动工业决策系统。Gold只由一名实际操作者审核，第二名领域复核属于后续增强；稀疏关键帧不足以单独证明完整动作，S04开机在严格视觉窗口中仍标记为不可见；Step Plan API承担部分语义与视觉推断，本地大模型部署尚未作为P0能力。何老师参考代码只完成静态分析，当前项目以自有N31闭环为主，不把上游Notebook历史输出说成本项目实跑结果。

下一步会继续扩展真实案例人工接受率评测，并在不改变Evidence边界的前提下提升开放式动作候选发现能力。对于正式赛事提交，团队资格、最终录屏、完整观看、真人彩排和官方规则仍由参赛者显式确认，任何程序都不会自动代签这些门禁。
