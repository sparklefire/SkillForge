# SkillForge 现场演示与录屏操作单

## 推荐顺序

1. PPT全屏打开，停在第1页。
2. 浏览器打开Web Demo，窗口宽度至少1280像素，页面缩放100%。
3. Web预先停在首屏，确认“GOLD / FINAL”和“严重错误5→0”能够出现。
4. PPT讲到40秒后切Web；110秒后展示成果；140秒切回PPT第7页。
5. 结束后不要继续操作，避免削弱结尾。

## DGX 稳定访问｜推荐

DGX 上一次性安装用户服务：

```bash
cd ~/skillforge/app
bash scripts/manage_dgx_demo_service.sh install
```

每次路演前在本机先跑烟测，再保持前台隧道：

```bash
bash scripts/dgx_demo_tunnel.sh --smoke
bash scripts/dgx_demo_tunnel.sh
```

浏览器打开 `http://127.0.0.1:17860`。该路径不依赖 DGX 公网端口，服务端只监听 `127.0.0.1:7860`；关闭隧道后本机入口立即消失。

## 三种运行模式

### 现场模式｜首选

```bash
bash scripts/run_demo_mode.sh live
```

- 现场重算操作者审核Gold的结构化受控错误集。
- 不发送原始素材，不调用外部模型。
- 预期：`GOLD / FINAL`，严重错误5→0，4项局部修订。

### 预处理模式｜技术问答时使用

```bash
bash scripts/run_demo_mode.sh preprocessed
```

- 复用已经通过隐私检查的安全派生素材。
- 重跑8来源本地摄取和Gold闭环，比现场模式慢。
- 用于回答“素材解析是否真实存在”，不建议占用三分钟主路演。

### 离线模式｜最终兜底

```bash
bash scripts/run_demo_mode.sh offline
```

- 只读取仓库内冻结的Gold演示包。
- 不含原始媒体和凭证，不依赖外部API。
- 网络、DGX或现场重算异常时立即切换。

三个命令均为原生Python方案，不要求Docker。

正式彩排前重新生成一次预处理结果并确认三种模式语义一致：

```bash
bash scripts/check_demo_mode_parity.sh --refresh-preprocessed
```

预期输出为 `PASSED`、3种模式、严重错误5→0、4项局部修订、网络请求0和外部模型调用0。该刷新严格使用本机OCR缓存，缺缓存时会安全失败而不是联网下载。此后开场前只需运行快速反向验真：

```bash
bash scripts/check_demo_mode_parity.sh --verify-only
```

## 开场前自动检查

```bash
bash scripts/check_project_board.sh
bash scripts/check_demo_mode_parity.sh --verify-only
bash scripts/check_submission_closeout.sh
bash scripts/check_pitch.sh
bash scripts/check_submission.sh
```

任务看板的 `AWAITING_HUMAN` 和 `AWAITING_EXTERNAL` 只描述正式提交待办，不等于开发目标 `BLOCKED`。7月19日基线应为 `ON_TRACK / implementation_goal_blocked=false`；到期任务未完成时工具会返回非零并列出安全任务ID。收尾状态器当前应显示 `TECHNICAL_READY_HUMAN_GATES_PENDING`、1/10阶段完成、0/5门禁确认，并把完整观看列为下一动作；返回2表示仍有安全待办，不是实现失败。

状态说明：

- `READY_FOR_SUBMISSION`：机器检查和五项门禁都已完成；提交预检返回0。
- `READY_WITH_HUMAN_GATES`：机器检查通过，但仍有人工/外部门禁；提交预检返回2。
- `DEVELOPMENT_CHECK`：开发模式跳过了Git干净状态、Git元数据或全量测试；提交预检返回2。
- `NOT_READY`：至少一项自动检查失败；提交预检返回1。

正式提交前唯一允许状态为 `READY_FOR_SUBMISSION`。

## 录屏建议

- 分辨率：1920×1080，横屏，30fps。
- 录制范围：只录PPT、Web和必要的播放器，不录终端中的环境变量或文件路径细节。
- 系统通知：关闭。
- 浏览器：关闭密码管理、下载记录和个人书签栏。
- 麦克风：先录10秒试听，确认无爆音和明显环境噪声。
- 时长：建议最终录屏控制在2分55秒至3分00秒。
- 备份：保留一份不含现场讲解的Web静默录屏，以便后期配音。

以上2分55秒至3分钟是项目内部彩排目标，不是尚未取得的官方视频时长规则。取得赛事讲义后应先更新规则结论，再决定是否调整成片长度。

## 80秒培训成片完整观看

必须从头到尾播放当前 `n31_training_video_v1.mp4`，不要拖动跳过。观看前初始化Git忽略的审核表：

```bash
bash scripts/check_training_video_review.sh --init
```

播放结束后编辑 `outputs/submission/training_video_review.json`：填写观看时间和播放方式，把状态改为 `READY_FOR_CHECK`。只有亲自确认以下八项后才能改为true：完整播放、旁白可听、旁白节奏可接受、音画同步、13步可理解、无敏感内容、无播放损坏、接受当前最终剪辑。

```bash
chmod 600 outputs/submission/training_video_review.json
bash scripts/check_training_video_review.sh
```

工具会重新核对当前成片、生成清单、自动QA和AI辅助抽帧检查，并生成不含私有备注或路径的 `training_video_review_qa.json`。机器通过只进入 `READY_FOR_HUMAN_CONFIRMATION`，不会自动关闭完整观看门禁；成片、生成清单或审核记录变化后旧QA和人工确认都会失效。

## 180秒真人计时彩排

彩排前先初始化Git忽略的私有记录；初始化不会覆盖已有记录：

```bash
bash scripts/check_final_rehearsal.sh --init
```

按冻结的7段运行单完成一次连续彩排，再编辑 `outputs/submission/final_stage_rehearsal.json`：

- 把状态改为 `READY_FOR_CHECK`，填写实际彩排时间、轮次、计时来源和总毫秒数。
- 每段填写连续的 `actual_start_ms / actual_end_ms`；首段从0开始，后一段起点必须等于前一段终点。
- 只有确实完成讲解、切屏操作、证明点核对和故障兜底准备后，才能把该段四项布尔值改为 `true`。
- 完整跑通、没有未恢复故障且没有展示敏感材料后，才能把三项总体验收改为 `true`。

然后运行：

```bash
chmod 600 outputs/submission/final_stage_rehearsal.json
bash scripts/check_final_rehearsal.sh
```

机器检查要求7段顺序和计划时间与当前运行单完全一致、实际时间连续、总时长处于内部175–180秒目标内。通过后生成0600权限的 `final_stage_rehearsal_qa.json`，只保存计数、时长和哈希，不复制私有备注或路径，状态仍为 `READY_FOR_HUMAN_CONFIRMATION`。该内部目标不代表尚未取得的官方视频规则。

## 最终录屏机器检查

录完后把最终候选文件保存在Git忽略的私有目录，固定文件名可以避免QA报告泄露个人命名信息：

```bash
mkdir -p outputs/submission
chmod 700 outputs/submission
cp "录屏软件导出的候选文件.mp4" outputs/submission/skillforge_final_recording.mp4
chmod 600 outputs/submission/skillforge_final_recording.mp4
bash scripts/check_final_recording.sh
```

机器检查覆盖：

- 私有目录0700、录屏文件0600且为固定MP4文件名。
- 内部目标时长175–180.5秒、1920×1080、29–31fps、H.264视频和AAC音轨。
- 音频采样率、综合响度、真峰值、累计/最长静音。
- 累计/最长黑屏。
- 报告不含媒体、绝对路径、凭证或外部模型调用，保存在 `outputs/submission/final_recording_qa.json`，权限0600。

通过后状态只会是 `READY_FOR_HUMAN_REVIEW`。字幕是否存在且清晰、是否出现真实面单/个人浏览器内容、是否完整播放、讲解与操作是否同步，均不能由机器结果代替。

## 发布冻结与上传后链接核验

最终录屏、PPT或其他18项成果发生变化后，先重新生成冻结清单；准备提交前只做严格复核：

```bash
bash scripts/build_release_manifest.sh
bash scripts/build_release_manifest.sh --verify-only
```

为录屏兜底或允许附件上传时准备统一技术交付包：

```bash
bash scripts/build_public_release_bundle.sh
bash scripts/build_public_release_bundle.sh --verify-only
```

只有两次命令均返回 `PASSED` 才使用该ZIP。它精确包含18项冻结成果、发布冻结清单和包内清单，不包含私有提交状态；包内仍标记五项人工门禁待完成。官方规则未确认附件格式前，只把它作为本地备份和技术交付候选，不直接上传。

上传完成后，初始化Git忽略的私有链接文件：

```bash
bash scripts/check_publication_links.sh --init
chmod 600 outputs/submission/publication_links.json
```

编辑 `outputs/submission/publication_links.json`，为 `PROJECT_PAGE`、`CODE_REPOSITORY` 和 `FINAL_RECORDING` 填入最终HTTPS网址，并把三项状态改为 `READY_FOR_CHECK`，然后执行：

```bash
bash scripts/check_publication_links.sh
```

检查器以匿名方式访问，不携带 `Authorization`、Cookie或其他凭证；带账号密码、片段、签名参数、密钥参数、私有主机或最终落到非公网IP的网址会被拒绝，并只允许HTTPS重定向。私有输入和QA报告权限均为0600、父目录0700且受Git忽略；QA只记录网址SHA-256、HTTP状态、内容类型、跳转次数和布尔结果，不复制真实网址、域名、IP或响应正文。只有三项均通过才表示“公开入口机器核验通过”，仍不能替代提交成功截图和人工打开复查。

## 提交成功回执与最终归档

全部人工门禁关闭且正式预检返回0后，先保存一份固定的最终预检依据：

```bash
bash scripts/check_submission.sh \
  --output outputs/submission/submission_preflight_final.json
```

提交成功、公开链接QA通过并再次打开作品页后，初始化私有回执审核表并复制截图或PDF：

```bash
bash scripts/check_submission_receipt.sh --init
bash scripts/check_submission_receipt.sh \
  --attach-receipt "提交成功截图.png"
```

编辑 `outputs/submission/submission_receipt_review.json`：填写带时区的提交时间和复核时间、提交编号或回执标识，把状态改为 `READY_FOR_CHECK`；只有亲自确认成功状态可见、项目身份一致、编号已记录、最终版本一致、作品页已重开、三个公开链接已复查、截图不含凭证后，才能把七项检查改为true。随后执行：

```bash
chmod 600 outputs/submission/submission_receipt_review.json
bash scripts/check_submission_receipt.sh
bash scripts/check_submission_receipt.sh --verify-only
```

工具要求最终预检必须是 `READY_FOR_SUBMISSION`、工作树干净、16项检查全部通过且无人工作业待办；发布清单仍匹配18项当前成果，公开链接QA必须通过、绑定当前私有网址输入并晚于提交时间。安全QA只保存哈希、数量、提交源码版本和布尔结论，不保存截图、提交编号、网址、个人信息或路径。当前尚未实际提交，因此这些私有文件必须保持不存在。

## 失败恢复

| 现象 | 立即动作 | 继续讲法 |
|---|---|---|
| Web打不开 | 启动offline模式；仍失败则切PPT第4页 | “这里是同一份冻结Gold审计结果” |
| 重跑按钮超时 | 不重复点击，展示已有前后对比 | “现场重算有离线同构兜底，审计内容不变” |
| 视频无法播放 | 展示PPT成果页和本地文件 | “成片已通过格式、覆盖和证据边界自动检查” |
| DGX不可达 | 使用PPT第6页的已保存计算报告 | “报告记录GPU、帧数、候选数和语义边界” |
| DGX公网端口不可用 | 保持SSH隧道；若隧道也失败则使用离线模式 | “演示服务默认不开放公网，现场入口是加密隧道” |
| 讲解超时 | 跳过测验细节，直接切结果页 | 保留5→0、证据、局部修订和结尾 |

## 人工门禁

- [ ] 参赛者完整观看80秒培训视频并确认旁白节奏。
- [ ] 按180秒脚本完成至少一次人工计时彩排。
- [ ] 完成有声字幕兜底录屏，并复核没有真实面单信息、密钥、绝对路径或个人浏览器内容。
- [ ] 确认报名队伍满足2–5人且最终成员、单位和分工一致。
- [ ] 按官方规则讲义核对必交材料、评分、视频时长、外部API和开源要求。

确认团队资格前，先生成并填写私有名单：

```bash
bash scripts/check_team_roster.sh --init
# 编辑 outputs/submission/team_roster.json，填2–5位成员并把状态改为 READY_FOR_CHECK
chmod 600 outputs/submission/team_roster.json
bash scripts/check_team_roster.sh
```

模板预列 `TECHNICAL_OWNER`、`EVIDENCE_OWNER`、`CONTENT_OWNER`、`DEMO_OPERATOR`、`SUBMISSION_OWNER` 和 `FINAL_REVIEWER` 六类职责，但不会生成姓名。机器通过只说明人数、资格声明、唯一主联系人和角色映射结构完整，状态仍为 `READY_FOR_HUMAN_CONFIRMATION`；随后还要对照报名页确认姓名、单位、2–5人和一人一队，再确认 `TEAM_ELIGIBILITY_CONFIRMED`。该确认绑定当时的名单SHA-256；名单缺失、失效或内容变化后确认自动失效，必须重新检查并确认。

取得官方规则原文后，先建立私有六项审核记录。优先复制官方PDF、网页快照或图片；如果只有稳定的官方网页，也可绑定不含账号、查询参数或片段的HTTPS网址：

```bash
bash scripts/check_official_rules_review.sh --init
bash scripts/check_official_rules_review.sh --attach-source "官方规则讲义.pdf"
# 或：bash scripts/check_official_rules_review.sh --source-url "https://官方域名/规则页面"
```

编辑 `outputs/submission/official_rules_review.json`，填写 `reviewed_at`，把状态改为 `READY_FOR_CHECK`；对 `SCORING_WEIGHTS`、`SUBMISSION_FIELDS`、`VIDEO_REQUIREMENTS`、`EXTERNAL_API_POLICY`、`OPEN_SOURCE_POLICY`、`ON_SITE_RUNTIME_REQUIREMENT` 六项分别填写结论与原文页码/章节/段落，确认后把每项 `confirmed` 和五个总检查都改为true。随后执行：

```bash
chmod 600 outputs/submission/official_rules_review.json
bash scripts/check_official_rules_review.sh
```

通过后会生成同目录0600权限的 `official_rules_review_qa.json`。QA只含审核表、公开快照和来源哈希、来源类型及布尔检查，不含规则正文或来源地址；状态仍为 `READY_FOR_HUMAN_CONFIRMATION / PENDING`，不能替参赛者确认。

先查看状态：

```bash
bash scripts/manage_human_gates.sh status
```

每项工作实际完成后再显式确认，不能提前确认，也不要直接修改 `cases/n31/pitch_runbook.json`。证据要求如下：完整观看必须绑定机器检查通过的私有审核记录；彩排必须绑定机器检查通过的私有计时记录；最终录屏绑定最终视频；团队资格绑定报名页截图或导出文件；官方规则必须绑定机器检查通过的固定私有六项审核记录，不能直接绑定任意讲义或网址。

```bash
bash scripts/manage_human_gates.sh confirm \
  --gate TRAINING_VIDEO_FULL_WATCH \
  --reviewer "确认人姓名" \
  --evidence-file outputs/submission/training_video_review.json \
  --note "已完整观看并确认旁白节奏"

bash scripts/manage_human_gates.sh confirm \
  --gate FINAL_STAGE_REHEARSAL \
  --reviewer "确认人姓名" \
  --evidence-file outputs/submission/final_stage_rehearsal.json \
  --note "已按冻结脚本完成连续计时彩排"

bash scripts/manage_human_gates.sh confirm \
  --gate FINAL_RECORDING_REVIEW \
  --reviewer "确认人姓名" \
  --evidence-file outputs/submission/skillforge_final_recording.mp4 \
  --note "已复核字幕、隐私、完整播放和音画同步"

bash scripts/manage_human_gates.sh confirm \
  --gate OFFICIAL_RULES_VERIFIED \
  --reviewer "确认人姓名" \
  --evidence-file outputs/submission/official_rules_review.json \
  --note "已按官方原文核对六项要求并更新提交方案"

bash scripts/manage_human_gates.sh revoke \
  --gate TRAINING_VIDEO_FULL_WATCH \
  --reviewer "撤销人姓名" \
  --note "成片更新，需要重新观看"
```

私有确认保存在Git忽略的 `outputs/submission/human_gate_confirmations.json`。确认器不复制证据内容；状态和预检不输出确认人、说明或证据路径。文件被修改、删除或运行单变化时确认失效，必须重新核验；五项有效确认和所有自动检查都通过后，本地干净预检才会返回 `READY_FOR_SUBMISSION`。

`TRAINING_VIDEO_FULL_WATCH` 只接受固定名称的本地审核记录，不接受视频本身或网址；同目录必须存在机器QA通过的报告，并绑定当前成片与生成清单。任一内容变化后都必须重新完整观看、检查和人工确认。

`FINAL_RECORDING_REVIEW` 还有额外保护：只接受本地文件，不接受网址；同目录必须存在机器QA通过的报告，且报告必须绑定同一录屏SHA-256和当前策略SHA-256。录屏或策略变化、QA报告缺失或失效时，都必须重新运行检查并再次人工确认。

`FINAL_STAGE_REHEARSAL` 同样只接受本地文件，不接受网址；同目录必须存在机器QA通过的报告，并绑定同一计时记录、当前7段运行单和内部彩排策略。任一内容变化后都必须重新检查和人工确认。

`OFFICIAL_RULES_VERIFIED` 只接受固定名称的本地六项审核记录，不接受讲义原件或网址本身；同目录必须存在机器QA通过的报告，并绑定当前公开核验快照和已复制来源文件或安全HTTPS网址的哈希。审核记录、QA、公开快照或本地来源变化后都必须重新检查和人工确认。
