## 结论

> 2026-07-17 更新：项目已完成公开赛事要求逐项核对。当前没有方向性偏离；2–5人组队资格、官方评分/提交/API细则仍待确认，DGX本地语义推理分量属于竞争力风险。详见 [赛事要求对齐矩阵](./docs/赛事要求对齐矩阵.md)。

如果目标是提高获奖概率，我建议做：

> **“匠传 SkillForge”——把专家操作视频、设备手册和口述经验，自动转化为可追溯、可验证、会自我修订的多模态培训包。**

例如输入一段设备维护视频、PDF 手册和工程师录音，Agent 自动生成：

- 分步骤 SOP
- 60 秒培训视频
- 图文安全卡片
- 手机端检查清单
- 多语种配音
- 培训测验
- 每条结论对应的视频时间点和手册页码

最关键的亮点不是“生成很多内容”，而是：**另一个质检 Agent 会检查步骤遗漏、危险操作、事实依据和画面一致性，然后自动重做不合格部分。**

这条方向同时覆盖了感知、理解、创作、持续优化，而且工业资料本地处理能够很好地解释“为什么必须用 DGX Spark”。

## 推荐项目怎么设计

核心演示最好只讲一个故事：

> 上传“老师傅更换某设备部件”的三分钟视频和安全手册，Agent 发现演示视频遗漏了断电确认步骤，拒绝直接发布；它引用手册依据，补齐警告画面，重新生成培训视频、海报和测验。

系统可以分成五个环节：

1. **感知 Agent**
   解析视频、关键帧、语音和 PDF，识别人物、工具、动作和安全信息。
2. **规划 Agent**
   将内容变成结构化步骤图：前置条件、操作动作、风险点、验证标准。
3. **创作 Agent**
   生成脚本、分镜、视觉素材、配音、字幕、清单和题库。
4. **质检 Agent**
   检查事实是否有来源、步骤是否完整、画面与文案是否一致、是否存在危险表述。
5. **优化 Agent**
   只重做低分片段；遇到重大歧义时请求人工确认，并保留修改记录。

视频解析可以直接借鉴 NVIDIA 面向 DGX Spark 的 VSS Blueprint，它已经包含视频摘要、搜索、问答、告警和事件复核能力，能显著缩短搭建时间。[NVIDIA VSS Blueprint](https://build.nvidia.com/spark/vss)

## 技术选型建议

- **核心大脑：Step 3.7 Flash 本地运行**
  它是原生视觉语言 MoE 模型，198B 总参数、约 11B 激活参数，支持工具调用、256K 上下文和不同推理档位，很适合长资料和多步骤 Agent。[Step 3.7 官方仓库](https://github.com/stepfun-ai/Step-3.7-Flash)
- **视频理解：NVIDIA VSS 或轻量自建流水线**
- **流程编排：显式状态机即可**
  不要为了“多 Agent”引入过重框架，重点是过程可观察、失败可重试。
- **媒体生成：比赛 API 或按阶段加载本地模型**
- **成片：FFmpeg**
- **界面：Gradio 或简单 Web UI**
- **评测：固定三组输入，比较质检前后的错误率**

一个必须注意的工程限制：DGX Spark 有128GB统一内存，官方定位可以运行约200B参数模型；但 Step 3.7 的 Q4 权重、视觉投影和运行开销已经接近整机内存上限。因此不要同时常驻 Step 3.7 和大型视频生成模型，应该采用**顺序加载**，或者让本地模型负责隐私资料理解与质检、API负责重型媒体生成。[DGX Spark 官方规格](https://www.nvidia.com/en-eu/products/workstations/dgx-spark/)

## 评委最容易记住的四个数字

不要只展示“效果很好”，至少准备：

- SOP 关键步骤覆盖率
- 有来源支撑的结论比例
- 质检前后严重错误数量
- 本地处理的数据比例、耗时和峰值内存

例如：首轮生成遗漏3项，质检闭环后降至0项；92%的操作结论能回溯到手册页码或视频时间点。数字可以小，但必须真实可复现。

## 不建议做的项目

- 普通多 Agent 聊天室
- 给提示词就生成一段视频
- 换壳式 RAG 知识库
- 同时做十种行业但每种都不深入

- 没有失败检测和自动修订的“流水线”

这些项目可能能运行，但难以证明“Agent 创作”以及 DGX Spark 的不可替代性。
